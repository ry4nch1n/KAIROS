# How GameRadar Runs & Is Maintained

The key mental model: **three things run on different clocks, and they are decoupled.**

```
  ① CRAWLER          ② DATA STORE            ③ DASHBOARD
  runs once/day  ──►  always there      ◄──  reads on demand
  (a cron job)       (the durable DB)        (a website / a file)
  writes new rows    never overwritten       renders charts
```

The crawler does **not** build the dashboard. The dashboard does **not** trigger the crawl. They only meet at the data store. This separation is what makes it cheap and reliable — if the website is down the data still collects; if a crawl fails the website still shows yesterday's data.

---

## Q1 — "Is it a newly generated HTML each run?"

**No — not in the recommended design.** The HTML is *not* regenerated by the crawl. There are three legitimate models; pick by how big the data gets and how much interactivity you want:

| Model | What runs each crawl | Dashboard is… | Hosting | Best when |
|---|---|---|---|---|
| **A. Static regenerate** | Crawl → write data → **rebuild a static site** (HTML + one JSON file) → redeploy | A folder of static files; all filtering happens in-browser on the JSON | Any static host (free) | **MVP / first months.** Simplest possible. |
| **B. Live app (recommended V1)** | Crawl → write to DB. *That's it.* | A running web app that **queries the DB live** when you open a page | Serverless web + managed DB | You have many filters, big data, network graphs, want it always current |
| **C. Hybrid (ISR)** | Crawl → write to DB → optionally ping a rebuild | Pages cached as static, **auto-revalidated** from DB on a timer | Serverless (Vercel) | Best of both — fast like static, fresh like live |

**Why "regenerate HTML every run" is usually the wrong default:** it couples the website to the crawl, rebuilds everything even when one number changed, and breaks down once you want server-side filtering/sorting over 18k games (you'd have to ship the entire dataset to the browser). It's fine as a *starting* point (Model A) precisely because the dataset is small early on.

**Recommendation:** start **Model A** (ship in a weekend, free), graduate to **Model C** when the dataset or your filtering needs outgrow shipping raw JSON to the browser. The data schema is identical for all three, so moving between them never touches your history.

## Q2 — "Should it be hosted on a server?"

**The dashboard: yes, but not a server you manage.** You almost never want a VPS you have to patch and keep alive for this. Ranked:

| Option | What it is | Cost | Ops burden | Verdict |
|---|---|---|---|---|
| **Vercel / Netlify / Cloudflare Pages** | Serverless web host, git-push to deploy | Free tier ample | ~none | ✅ **Use this.** |
| **Supabase / Neon** (for the DB) | Managed Postgres, auto-REST API | Free tier (then ~$0–25/mo) | ~none | ✅ **Use this** for the data store |
| **Single small VPS** (Hetzner/Fly $5/mo) | One box runs DB + web + cron | ~$5/mo | You patch/monitor it | Only if you want everything in one place |
| **Your workstation** | Run locally | Free | PC must be on; dies if PC dies | ❌ Not for the durable parts |

**Important for your multi-PC setup:** the *durable* parts (DB + the daily crawl) must live in the cloud, **not** on any one workstation — you've said you may lose any machine. A managed DB + cloud cron means losing a PC costs you nothing. (You can still *view* the dashboard locally, or even run a local copy for dev.)

## Q3 — "Should there be a cron job for the daily crawl?"

**Yes — a scheduled job in the cloud, independent of your PCs.** Options, best first:

| Scheduler | How | Pros | Cons |
|---|---|---|---|
| **GitHub Actions cron** ⭐ | `on: schedule: - cron: "0 4 * * *"` in a workflow that runs the Python crawler | Free, serverless, version-controlled, survives PC loss, logs built-in, can commit data or push to DB | 6h job limit (irrelevant), needs secrets stored in repo settings |
| **Supabase Edge Function + pg_cron** | DB-native schedule | Lives next to the data | Heavier crawls (Playwright) don't fit edge runtime well |
| **Fly.io / Render cron** | Container on a schedule | Full runtime, good for Playwright | A bit more setup |
| **Windows Task Scheduler** (your workstation) | Local `.bat`/PowerShell | You already use this pattern | ❌ PC must be on; not portable — **avoid for the crawl** |

**Recommended: GitHub Actions.** Concretely:

```yaml
# .github/workflows/crawl.yml
name: Daily Crawl
on:
  schedule:
    - cron: "0 4 * * *"      # 04:00 UTC daily
  workflow_dispatch:          # lets you trigger a run by hand
jobs:
  crawl:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with: { python-version: "3.12" }
      - run: pip install -r crawler/requirements.txt && playwright install chromium
      - run: python -m crawler.run_crawl
        env:
          DATABASE_URL: ${{ secrets.DATABASE_URL }}      # points at Supabase
      - run: python -m enrich.run_enrich                  # AI pass (V1+)
        env:
          DATABASE_URL: ${{ secrets.DATABASE_URL }}
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
```

The crawl writes rows to Supabase and exits. The website (Vercel) reads those rows whenever someone opens it. No regeneration, no coupling.

## The recommended end-to-end setup (concrete)

```
GitHub Actions (cron, daily 04:00)
        │  runs Python crawler + enrichment
        ▼
Supabase Postgres  ◄─── the only durable thing; append-only history
        ▲
        │  live queries (or ISR revalidate)
Vercel (Next.js dashboard)  ◄─── you open this in a browser, anytime
```

- **Cost:** $0 to start (all free tiers); ~$0–25/mo once data/traffic grows.
- **Survives losing any PC:** everything durable is cloud-side.
- **Always current:** dashboard reflects the latest crawl with no manual step.
- **One thing to deploy:** a daily GitHub Actions job ships `main` to Netlify if (and only if) it changed since the last deploy — the crawl is just another workflow in the same repo.

### MVP with a real database (recommended)
The MVP uses a **managed Postgres from day one** — no SQLite-in-repo throwaway. Concrete free-tier shape on a Netlify account:

```
GitHub Actions (daily cron) ─► Neon Postgres (serverless, free, scales to zero)
                                      ▲
        Netlify (Next.js dashboard) ──┘  reads via Netlify Functions / server components
```

- **Database: Neon** (serverless Postgres). Why Neon for this MVP: native **Netlify DB integration** (one click from the Netlify dashboard provisions a Neon DB and wires the connection string), generous free tier, scales to zero when idle so it costs nothing between crawls, pure Postgres so every query in §2 of DESIGN.md works unchanged. **Supabase** is the alternative if you also want built-in object storage for cached thumbnails + an instant auto-REST layer — pick it only if you want those batteries now.
- **Crawl writes to Neon**, doesn't touch the website. **Dashboard reads from Neon** live (or ISR-cached). Same decoupling as before.
- **Thumbnails in MVP:** store the source CDN `thumbnail_url` and display it directly — defer caching to object storage until V1. Keeps the MVP to one external service (the DB).
- The schema, adapters, and dashboard all carry forward to V1 unchanged — adopting the DB now means you never do a data migration later, and your history starts accumulating in its permanent home on day one.

**Connection wiring:** put the Neon connection string in two places — a **GitHub Actions secret** (`DATABASE_URL`, used by the crawl) and a **Netlify environment variable** (used by the dashboard). Use Neon's *pooled* connection string for the serverless dashboard, the direct string for the batch crawl.

## Maintenance (what you actually do over time)

| Cadence | Task | Effort |
|---|---|---|
| **Never (automatic)** | Daily crawl, snapshot append, view refresh; PRs auto-merge on green CI; a daily job deploys `main` to Netlify only if it changed (the `live` tag marks what's in production) | 0 |
| **When a site changes layout** | An adapter breaks → crawl logs `partial` + alerts you → fix *one* adapter file, update its test fixture | ~30 min, rare |
| **When you improve the AI prompt** | Bump `prompt_version` → enrichment re-runs only affected games next crawl | 0 after the edit |
| **Occasionally** | Add a new source (itch/Steam) = write one new adapter class | a few hours, optional |
| **Rarely** | Schema change = numbered SQL migration | minutes |

Two habits keep it self-healing: **(1)** record HTML/JSON fixtures so adapter tests catch breakage offline, and **(2)** have the crawl push a one-line status to wherever you'll see it (Lark/Notion/email) so a silent failure can't rot for weeks. Because history is append-only, a bad crawl is deleted by its `crawl_id` without corrupting anything.

## Command center (KAIROS) — how the three services run together

KAIROS is **one web app, one Neon DB, one Netlify deploy** — the three services differ only by route group and which tables they read.

```
GitHub Actions (daily) ─► radar.*  ┐
Mon/Thu brief routine  ─► brief_editions ├─► Neon ─► Netlify (KAIROS shell: /radar /brief /library)
(future) you           ─► library_items ┘
```

| Service | Data source | Runs / updated by |
|---|---|---|
| GameRadar | `radar.*` (games/snapshots) | the daily crawl cron (GitHub Actions) |
| News Brief | `brief_editions` | your existing Mon/Thu routine + **one new write-step** |
| Library | `library_items` | you, in V2 |

**News Brief write-step (the only change to your current routine):** after it builds an edition and writes the local HTML + Notion copy as it does today, add a final call that upserts the structured edition into `brief_editions` (and optionally the rendered HTML). Because the brief routine may run on a workstation while the crawl runs in GitHub Actions, both simply need the Neon `DATABASE_URL` — they write to the same DB independently, on their own schedules. Nothing about the existing local-HTML or Notion output changes.

**Local dev vs prod parity:** locally the whole command center runs against PGlite with one `npm run dev`; in prod the identical SQL/handlers run against Neon on Netlify. You verify locally, deploy unchanged.

## Phase 2 — Steam (PC) crawl (added 2026-06-30)

```
# crawl real Steam data into the DB (PGlite locally, or Neon if DATABASE_URL set)
cd app && CRAWL_LIMIT=150 npm run crawl:steam

# live end-to-end validation (in-memory; prints tier distribution + indie genre economics)
cd app && STEAM_VALIDATE_LIMIT=18 npx tsx server/scripts/validate-steam.ts
```

- No API keys. Three free endpoints per app (appdetails + appreviews + SteamSpy), throttled ~1.5s/app.
- **Seed is indie-aware** (`mergeSeeds` round-robins SteamSpy `tag=Indie` + `top100in2weeks` + storefront featured) so the sample isn't all-AAA.
- **SteamSpy rate limits:** the `all` endpoint is 1 req/min — *not used*; only `top100in2weeks` + per-app `appdetails` (fine). Keep `CRAWL_LIMIT` modest to stay polite and inside the GitHub Actions free budget (private repo) — Steam adds ~10–15 min/day at ~150 apps.
- Schema migration is additive + idempotent: `npm run db:migrate` (or the seed/crawl applies schema locally) adds the new columns on existing Neon without data loss.

### Phase 2 — viewing Steam analytics in the app
1. `cd app && npm run db:seed` then `CRAWL_LIMIT=40 npm run crawl:steam` (populate local DB; prod uses the daily crawl).
2. `npm run dev` → open http://localhost:5173 → GameRadar → click the **Steam** platform tab.
3. The Steam tab reads `GET /api/steam`; toggle the genre-economics cohort between **Indie** (default) and **All tiers** (demand context). Steam data shows only after a Steam crawl has populated the DB.

### Daily crawl — Actions budget (private repo)
`crawl.yml` runs daily (04:00 SGT): **migrate (idempotent schema) → CrazyGames 250 → Poki 250 → Steam 200**, all into Neon. Free private-repo tier = 2,000 Actions min/month. **Measured** (run 28423375066): CrazyGames 250 ≈ 19 min, Poki 250 ≈ 20 min, Steam ≈ 5.6 s/app → 200 ≈ 19 min, +overhead ≈ 2 min ⇒ **≈ 60 min/run → ~1,800/mo (30 runs) / ~1,860 (31)**, under cap with ~150–200 min headroom. (The first manual run used Steam 300 ≈ 28 min → ~68 min/run ≈ 2,050/mo, over cap; reduced to 200.) The migrate step ensures the Phase 2 Steam columns exist before the Steam crawl. To go bigger, make the repo public (unlimited Actions) and raise the per-source `CRAWL_LIMIT`s — no secrets live in code.
